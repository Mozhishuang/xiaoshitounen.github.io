Java面试- == & equal & hascode

发表于2020-04-11|更新于2020-06-04|Java
|阅读量:

一. == 和 equal的区别

1. == 比较的是值是否相等

如果作用的是基本数据类型的变量，则直接比较其存储的值是否相等。

如果作用的是引用类型的变量，则比较的是所指向的对象的地址。

2. equal

equal() 方法继承于 Object 类，比较的是否是同一个对象，它不能作用于基本数据类型的变量。

如果没有对 equal() 方法重写，则比较的是所指向对象的地址。

诸如 String，Data 等类对 equals 方法进行了重写，比较的是指向对象的内容。

若比较的是两个独立的对象，则总返回false。如果编写的类希望能够比较该类创建的两个实例对象的内容是否相同，那么必须覆盖equals()方法，由开发人员自己编写代码来决定在什么情况下即可认为两个对象的内容是相同的。

二. hasCode 和 equal

在 JDK 中，Object 的 hashcode 方法是本地方法，也就是用 c 语言或 c++ 实现的，该方法直接返回对象的 内存地址。

1. hasCode

哈希码并不是完全唯一的，它是一种算法，让同一个类的对象按照自己不同的特征尽量的有不同的哈希码，但不表示不同的对象哈希码完全不同。hashCode() 也可以用来对比两个对象是否相等一致。

2. 为什么有了 equal 还需要 hasCode

因为重写的equal（）里一般比较的比较全面比较复杂，这样效率就比较低，而利用hashCode()进行对比，则只要生成一个hash值进行比较就可以了，效率很高。

3. 为什么有了 hasCode 还需要 equal

因为hashCode()并不是完全可靠，有时候不同的对象他们生成的hashcode也会一样（生成hash值得公式可能存在的问题），所以hashCode()只能说是大部分时候可靠，并不是绝对可靠。

4. hasCode 和 equal 之间的关系

① equal 相同的 两个对象，其 hasCode 一定相同。

② equal 不相同的 两个对象，其 hasCode 可能相同也可能不同。

③ hasCode 不相同的 两个对象，其 equal 必定不相同。

④ hasCode 相同的 两个对象，其 equal 可能相同也可能不同。

5. 总结

① 如果重写了equal 方法，必须重写 hashCode 方法，原因是默认的 hashCode 是将对象的存储地址进行映射。而且逻辑上，如果两个对象的 equal 方法返回是相等的，那么它们的 hashCode 必须相等；反之不一定成立。

② 一般情况下，重写 equal 方法，第一步是比较 hashCode 值。一旦 hashCode 值不相同也就不需要继续比较下去。
